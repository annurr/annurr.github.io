# Architecture Schema (Public View)
[Frontend: GitHub Pages | Backend: Supabase/PostgreSQL]

## 1. High-Level Topology
The portfolio system operates on a detached, serverless backend.
- **Client Application:** Statically hosted on GitHub Pages. Retrieves content entirely via REST/SDK APIs without containing internal secrets or logic.
- **Backend Database:** Powered by PostgreSQL and Supabase. Enforces strict schema rules. 
- **Row-Level Security (RLS):** All tables are heavily locked down. The frontend relies exclusively on the public `anon` key, inherently incapable of executing destructive actions, relying entirely on JWT auth tokens to prove administrative status.

## 2. Protected Monitoring Services
- **GitHub Action (`.github/workflows/keep_alive.yml`)**: An entirely external, recurring service that executes API pings to prevent backend hibernation. Results are logged to dedicated tables.
- **Internal System Heartbeats**: The database enforces localized internal chronometric transactions via `pg_cron` entirely circumventing the public perimeter to monitor uptime integrity. 

## 3. UI Component Lifecycle
- **Dynamic Render DOM:** All administrative dashboards operate dynamically. Switching views (such as the *Audit Logs* or *System Monitor*) actively flushes internal tables and dynamically suppresses unused views to preserve execution memory and prevent phantom "attached" UI nodes.

*Note: For granular schema definitions, exact primary keys, and internal policy rules, please reference the private `.github/architecture_schema.md` within the administrative repository root.*

## 4. Premium Tools Authentication Check
- **Gated Elements:** The `tools.html` module contains premium utilities physically present in the DOM but locked via CSS overlays (`backdrop-filter`). Unlocking requires an active JWT admin session from the Supabase client, resolving securely entirely on the client-side execution block without triggering full page re-renders.
