# Annurr Portfolio & Blog - Architecture Schema

This project is a dynamic portfolio and blog management system designed to be lightweight but robust.

## 1. Core Technologies
- **Frontend Hosting:** GitHub Pages
- **Backend & Database:** Supabase (PostgreSQL)
- **Frontend Stack:** Vanilla HTML, CSS, JavaScript (No heavy JS frameworks)

## 2. Architecture Diagram

[ Frontend (GitHub Pages) ]
  |-- index.html : Main portfolio page. Dynamically fetches its text and layout configuration from Supabase.
  |-- blog.html  : Public blog page. Fetches and displays published blog posts from the database.
  |-- admin.html : Secure dashboard to manage posts, edit homepage config, and view frontend audit logs.
  |-- /assets    : Static files, schemas, and icons.
  |-- /js & /css : Scripts and stylesheets for application logic and UI.

       |  (REST API / fetch via supabase-js or standard HTTP requests)
       v

[ Backend (Supabase) ]
  |-- Authentication : Protects the admin panel. Only authorized users can access the dashboard.
  |-- PostgreSQL Database :
       |-- `homepage_config` : Stores dynamic portfolio settings (hero text, links, etc).
       |-- `posts`           : Stores blog content (title, excerpt, body, image_url).
       |-- `audit_logs`      : Custom logging table recording frontend admin actions (logins, creations, deletions).
       |-- `heartbeat_logs`  : Automated table for database uptime monitoring.
       |-- `github_action_logs` : Stores keep-alive ping statuses triggered by GitHub Actions.

## 3. Security & Monitoring
- **Row Level Security (RLS):** Enforced on Supabase tables ensuring that only authenticated admin users can Insert/Update/Delete data (public can only Select/Read).
- **Keep-Alive (Anti-Pause):**
  - **GitHub Actions:** `.github/workflows/keep_alive.yml` runs a daily `curl` ping against the Supabase REST API `/rest/v1/`.
  - **Supabase pg_cron:** An internal job runs daily to insert a timestamp into `heartbeat_logs` to keep the postgres instance active.
